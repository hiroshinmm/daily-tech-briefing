# プロジェクト比較調査 結果まとめ (Walkthrough)

「Ice Break」と「HighYield\weekly-icebreak-email」の比較調査を完了しました。以下にその詳細をまとめます。

## 1. アーキテクチャと設計思想

| 特徴 | HighYield (旧) | Ice Break (現) |
| :--- | :--- | :--- |
| **設計構造** | モノリス (`index.js` に集約) | モジュール設計 (機能ごとに分離) |
| **メンテナンス性** | 密結合のため修正が他に影響しやすい | 疎結合のため部分的な改善が容易 |
| **実行環境** | ローカルPC (Windows) | GitHub Actions (Ubuntu) |
| **自動化の安定性** | PCの電源状態に依存 | クラウド上で確実に実行 |

---

## 2. 情報ソースと記事選別

*   **HighYield**:
    *   固定のRSSフィードを巡回。
    *   キーワードの有無による単純なフィルタリング。
    *   カテゴリ数: 5
*   **Ice Break**:
    *   Google News検索RSSを活用した動的な情報収集。
    *   **Geminiによる知的な選別**: 10件の候補から最適な1件をAIが選択。
    *   カテゴリ数: 10 (事業領域に最適化)

---

## 3. GitHub Actions パフォーマンスとコスト

### 実行時間の違い
*   **Ice Break (約4〜6分)**: 
    *   より多くのカテゴリ処理、Puppeteerによる高画質キャプチャ、GitHub Pagesへのデプロイを含むため。
*   **HighYield (約2分相当)**: 
    *   最小限の処理のみ。

### AIトークン消費量
*   **Ice Break**: HighYieldの**約20倍**。
    *   理由: 選別用に全候補記事（100件分）をAIに読み込ませているため。
*   **コスト感**: Geminiの最新低価格モデル（Flash-lite等）の使用により、実行コスト増は微々たるもの（1回数円程度）。

---

## 4. AIモデルの選定方法と推論プロセス（詳細）

AIモデルの選定においては、単なる「性能」だけでなく、**「運用コスト」「データの構造化」「文脈理解量」**のバランスが重要になります。

### A. Ice Break の選定基準（クオリティと安定性重視）
Ice Breakでは `gemini-3.1-flash-lite-preview`（または最新のFlash系）を主軸に置いています。

1.  **長いコンテキスト窓の活用**:
    *   カテゴリごとに10件の記事（計100件程度）を一度にプロンプトに流し込みます。これらを「俯瞰」して比較選別するためには、一定以上の文脈理解能力と、長い入力を安価に処理できるFlashモデルの特性が不可欠です。
2.  **JSON Mode の堅牢性**:
    *   後続のスライド生成やメール送信がプログラムで自動化されているため、1文字でも形式が崩れるとシステムが止まります。最新のGeminiモデルは `responseMimeType: "application/json"` によるスキーマ制御が非常に強力で、出力の安定性が選定の決め手となっています。
3.  **推論の「厚み」**:
    *   HighYieldが「翻訳」を主目的とするのに対し、Ice Breakは「エンジニアとしての洞察（Insight）」を求めます。これには、技術用語の理解が深いモデル（学習データが最新のもの）である必要があります。

### B. HighYield の選定基準（可用性と回避重視）
HighYieldでは、特定のモデルに依存せず「確実に止まらないこと」を重視したリスト形式（Waterfall方式）を採用しています。

1.  **レートリミット（429エラー）対策**:
    *   無料枠での利用を想定し、1つのモデルが制限にかかった瞬間に `gemini-3.1-flash-lite` → `gemini-3.0-flash` → `gemini-2.5-flash` ・・・と自動で切り替わります。
2.  **目的の単純化**:
    *   1つの入力に対して1つの翻訳を出すだけなので、モデルごとの「賢さ」の差よりも、APIが応答を返す「生存率」を最優先しています。

### C. なぜ「Pro」ではなく「Flash」なのか
両プロジェクトとも、最高性能の `Gemini Pro` ではなく `Flash` を選択しています。

*   **速度**: デイリー実行において、Proの数倍速いFlashはタイムアウトのリスクを減らします。
*   **コスト**: Ice Breakのように「選別」のために大量のテキスト（トークン）を流す場合、Proだとコストが跳ね上がりますが、Flashならほぼ無視できる（または無料枠に収まる）コストで運用可能です。
*   **十分な賢さ**: ニュースの選別と要約というタスクにおいて、現在のFlashモデルはProに匹敵する精度に達しています。

## 5. その他の技術的な実装の違い

| 機能 | HighYield (旧) | Ice Break (現) |
| :--- | :--- | :--- |
| **テンプレート** | JavaScript内にHTML文字列をハードコード | **EJS (Template Engine)** を採用し、ロジックとデザインを完全分離 |
| **データ管理** | 一時的な変数・配列のみで処理 | `data/` フォルダに **JSON形式で中間データを保存**。デバッグや再利用が可能 |
| **画像生成** | 記事のOGP画像をそのまま引用がメイン | **AI生成プロンプト**を元に、UnsplashやAI生成による最適な画像を動的に決定 |
| **UI/UX** | メールのみの通知 | **Responsive Web (GitHub Pages)** + リッチなHTMLメールのハイブリッド |

---

## 総評（追加）
「Ice Break」は、AIを単なる「翻訳機」としてではなく、**「編集者（エディター）」として活用**している点が最大の技術的進化です。また、EJS等のモダンなWeb技術を取り入れることで、デザインの修正や機能拡張が簡単に行える「開発基盤」としての完成度が高まっています。
